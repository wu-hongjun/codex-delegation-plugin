## CRITICAL DISCOVERY — schema mismatch with real Claude Code 2.1.149

T12 is paused after `setup`. The setup probe surfaced two failures and the diagnostic deep-dive that followed revealed substantial contract differences between the assumed schema (the mock + research report) and real Claude Code 2.1.149.

### Probe results

- node-version, codex-version, claude-binary, claude-version, claude-auth, claude-agents-json, claude-logs, transcript-path, codex-plugin-trust → OK
- **claude-bg-flag → FAIL**: `claude --help` does not list `--bg` (real Claude hides it from help output)
- **claude-daemon → FAIL**: `claude daemon status` subcommand does not exist in 2.1.149

### Direct probes against real Claude

1. `claude --bg "test"` (despite not being in --help) actually WORKS:
   ```
   Starting background service…
   backgrounded · 8f7f2405
     claude agents             list sessions
     claude attach 8f7f2405    open in this terminal
     claude logs 8f7f2405      show recent output
   ```

   The orphan session 8f7f2405 was immediately stopped with `claude stop 8f7f2405` (verified successful).

2. **stdout format mismatch**: real Claude prints `backgrounded · <shortId>` not `Started background session <shortId>`. Our `parseShortId` (background-session.ts) looks for the 'session' keyword token then falls back to last digit-containing token. Real output's 'backgrounded' would skip Strategy 1; the fallback regex `/^(?=[a-zA-Z0-9_-]*\d)[a-zA-Z0-9_-]{4,}$/` *would* match `8f7f2405` so the parser MIGHT still work, but the contract isn't formal.

3. **agents --json schema differs significantly** from our assumed mock shape:
   Real fields per entry: `pid, cwd, kind, startedAt (UNIX number), sessionId (UUID), status`
   Mock-assumed fields: `id, shortId, sessionId, name, cwd, pid, status, startedAt (ISO string), updatedAt, transcriptPath`
   - No `shortId` or `id` field — shortId is only emitted by `claude --bg` stdout (e.g. `8f7f2405`) and is derived from the UUID first segment minus dashes (`8f7f2405` matches sessionId `8f7f2405-...`)
   - No `name` field — so our session-name matching strategy can't work
   - No `transcriptPath` field
   - No `updatedAt` field
   - `startedAt` is a Unix epoch number, not ISO string

4. **Status values differ**: real values seen include `idle, busy, waiting`. Our normalizer maps `working/idle/needs_input/completed/failed/stopped/orphaned/unknown`. `busy` would map to `unknown`, `waiting` would map to `needs_input`.

5. **Session-handle matching breaks**: our reconciler matches by `sessionId` (full UUID) OR `shortId` OR `sessionName`. With real Claude:
   - `sessionId` is a UUID we don't yet have when we get the shortId back from --bg
   - `shortId` isn't in agents --json output (only the UUID)
   - `name` (sessionName) doesn't exist in real output
   - Practical match path: shortId (8 hex) is the prefix of `sessionId.replace(/-/g, '').slice(0,8)`. Reconciler needs this derivation rule.

### Impact

The static test suite (335 tests, all green) passes against the MOCK. The mock was built to a schema that doesn't match real Claude Code 2.1.149. T12's live E2E reveals this. None of T2–T11 is wrong against the mock; what's wrong is the mock itself — and any driver code that depended on mock-specific fields will need to learn the real schema.

Re-run all tests. This is substantial — likely 1-2 task-equivalents.

The Block A1 finding ('Claude Code now exposes first-party background-session management') was directionally right but the exact shape of `claude agents --json` and `claude --bg` output was incorrect in the report. May require a follow-up research pass before code rework.

## Key finding: `idle` ≠ `completed` for interactive background sessions

Real Claude reported the session as `idle` while our dispatcher reported `running`.
Inspection of the underlying TUI logs (claude logs <id>) shows the agent ALREADY
completed the task — produced the right answer:

    Found 3 TODOs in the repo:
      - README.md:3 — TODO: replace this placeholder
      - README.md:4 — TODO: another todo here
      - app.js:2  — TODO: implement

This is correct behavior on the real Claude side: `idle` means 'agent has output
and is awaiting the next user input', not 'finished'. Our T9 reconciler maps
driver-`idle` → job-`running` (T9 spec line 'idle → running').

**Semantic gap surfaced**: for an interactive background session that the user has
no plans to send follow-up input to, `idle` is effectively a terminal state.
Our dispatcher cannot distinguish 'still working' from 'awaiting next user input'.

Two reasonable fixes (deferred to T12b or follow-on plan):
1. T9 reconciler maps driver-`idle` → job-`needs_input` instead of `running`.
   Then `result` would honor 'awaiting input' as a soft-terminal state and print
   the assistant message that's available so far. This matches OQ1 (user manually
   attaches to provide further input or stop the session).
2. Inspect transcript to detect 'last event was an assistant message' as the
   real completion signal. Real Claude transcripts are ANSI-encoded TUI replays,
   not JSON event streams — so T8's transcript reader doesn't extract events from
   them. This is a deeper rework (likely Plan 0002 or schema-research follow-up).

**Also discovered**: real Claude reports `kind: 'background'` for sessions
started via `claude --bg`; our mock reports `kind: 'interactive'`. Cosmetic
difference, not load-bearing.

## Known limitations from this E2E (deferred)

1. **`idle` ≠ `completed`** for interactive background sessions: Claude session goes `idle` (awaiting next input) when the agent finishes turn. The T9 reconciler maps driver-`idle` → job-`running`, so `result` won't auto-fire on a quiet session. Working pattern: user issues `stop`, then `result` extracts the message. Future polish: map `idle` → `needs_input` (with attach hint) or add a `--wait-for-idle` flag on delegate.
2. **Real Claude transcripts are ANSI TUI byte streams**, not JSON event records. Our T8 transcript parser can't extract events from them. The orphan→logs fallback path correctly recovered the message text post-stop. For interactive resume / mid-turn streaming, transcript parsing would need real-schema rework (Plan 0002 or follow-on).
3. **`kind: 'background'` (real) vs `'interactive'` (mock)**: cosmetic; not load-bearing.

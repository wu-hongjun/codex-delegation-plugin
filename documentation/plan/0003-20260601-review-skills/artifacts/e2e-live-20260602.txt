BUG OBSERVED during first review attempt: T12a remediation
==================================================================

Two interlocking bugs surfaced during the live E2E (only Plan 0002 code
paths were affected — both are pre-existing issues that Plan 0003's
review workflow happens to expose first):

  1. `CC_PLUGIN_CODEX_ATTACH_WARMUP_MS` advertised in attach.ts as an
     operator-tunable env var was UNREACHABLE from the production
     dispatcher path. attach.ts read it from `opts?.env?.[…]`, but the
     dispatcher constructs `new ClaudeBackgroundDriver({ cwd: workspace })`
     without an `env` option — so `opts.env` was always undefined and the
     env var fell through to the 2000ms default no matter what the user
     set in their shell.

  2. Bare CR (`\r`) at end of prompt write did not submit on Claude Code
     2.1.150 (it worked on 2.1.149). Even when the warmup was extended,
     long prompts (≥ ~few hundred bytes) were never recognized as input.
     The Claude TUI swallowed them. Probe evidence:
        - warmup=8s + raw "What is 2+2?\r\n":   tempo flips to active
                                                within 1s (short prompt
                                                survives the raw path).
        - warmup=8s + raw 1574-byte review:    tempo stays idle for 15s+
                                                even though >24000 bytes
                                                of TUI output are received.
        - warmup=8s + bracketed-paste-wrapped 1574-byte review:
                                                tempo flips to active
                                                within ~2s. Submitted.

T12a remediation (applied in this T12 commit per the brief's
"if T12 exposes a bug" process):

  - attach.ts: env-var read for `CC_PLUGIN_CODEX_ATTACH_WARMUP_MS` and the
    new `CC_PLUGIN_CODEX_PROMPT_REGISTER_TIMEOUT_MS` falls back to
    `process.env` when `opts.env` is absent. Test seam (opts.env) takes
    precedence; production path now reaches the env vars.
  - attach.ts: ATTACH_WARMUP_DEFAULT_MS bumped 2_000 → 8_000ms. Live
    probe with Claude 2.1.150 required 5_000ms+ reliably; 8_000ms gives
    margin and still completes well inside the prompt-register window.
  - attach.ts: prompt write encoded as
        '\x1b[200~' + input.text + '\x1b[201~\r'
    (bracketed-paste wrap + CR) instead of bare CRLF. Permission-answer
    write uses the same encoding.
  - tools/mock-claude/claude: input handler strips `\x1b[200~`/`\x1b[201~`
    markers from accumulated buffer before submitting, and skips empty
    submits (CRLF terminators in icrnl-line-discipline mode would
    otherwise fire a second empty submit and clobber the response).
  - 3 new regression tests in `packages/driver-claude-code/test/send.test.mjs`:
        T12a-A: env-var fallback to process.env when opts.env undefined
        T12a-B: CC_PLUGIN_CODEX_PROMPT_REGISTER_TIMEOUT_MS reachable via
                process.env (1ms override forces fast timeout)
        T12a-C: attach.ts source uses ESC[200~ … ESC[201~ + CR (not bare
                CR or CRLF) for both prompt and permission-answer writes

Test count delta: driver lane 175 → 178 (+3). All four lanes green
post-fix; total 1010/1010 (was 1007). `test:attach` 26/26 (was 25).

(Footnote on the NIT: the dispatcher passes `sendResult.finalMessage`
to the structured-output parser. The driver populates that from sidecar
`output.result`, which on Claude 2.1.150 ends up as a short SUMMARY
string ("review verdict: pass — all TODOs found, no omissions") rather
than the full assistant message containing the fenced JSON block. The
underlying reconciler subsequently reads the full message from logs
and writes it to <jobId>.result.md and to turn[1].result.finalMessagePreview
— verified post-stop in STEP 9. So the parseable structured findings ARE
captured by the job store; only the in-flight `cmdReview` output uses
the sidecar summary. Documented here for the auditor; not a Plan 0003 T12
remediation candidate — it's a Plan 0002 driver-vs-reconciler split that
belongs to a future plan.)

Root cause:
  packages/plugin-codex/ depends on workspace packages
  @cc-plugin-codex/runtime and @cc-plugin-codex/driver-claude-code.
  Source path resolves these via the workspace's root node_modules.
  Marketplace tree (committed under marketplace/plugins/claude-companion/)
  is byte-identical to source but contains NO node_modules. When Codex
  installs the plugin into <CODEX_HOME>/plugins/cache/.../0.2.0/, that
  cache copy also has no node_modules. The dispatcher CAN load when
  invoked from the source path (workspace resolves), but FAILS when
  invoked from the cache path (no resolution path to workspace deps).

Skill-discovery scope:
  Codex's TUI $<skill-name> expansion works for all 8 skills (qa-tester
  PASS confirmed). This finding does NOT invalidate Plan 0006's skill
  registration / marketplace allowlist / packaging-check work.

Skill-execution scope:
  Every dispatcher-backed skill is blocked by this missing dep when run
  from the cache. That is the user-facing behavior after a real
  `codex plugin add` against the local marketplace. The discovery
  layer (qa-tester PASS for 7/8) is the floor, not the ceiling, of
  what users will experience.

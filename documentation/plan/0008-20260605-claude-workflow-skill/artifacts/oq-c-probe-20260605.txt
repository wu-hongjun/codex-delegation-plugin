Evidence: `"num_turns":1`, `"result":"OK"`, `"output_tokens":6`. Claude returned the literal word "OK" in a single turn with no workflow planning, no multi-phase structure, and no approval dialog. The `ultracode:` prefix was treated as a plain text prompt prefix; no workflow engine was invoked.

Supporting signals:
- `fast_mode_state":"off"` — fast/workflow mode was not engaged
- `"iterations"` array has exactly one entry (single model call, no subagents)
- No workflow-related keys in the JSON (no `workflow_id`, no `phases`, no `agents`)

**Conclusion for Plan 0008**: `claude -p "ultracode: ..."` does NOT trigger workflow planning in print mode. The `ultracode:` keyword is a TUI-only trigger. The Plan 0008 design correctly uses `claude --bg` (not `claude -p`) for the `$claude-workflow` skill, which is the right path.

**Backlog implication (per plan §8)**: A future `$claude-workflow --print` mode (non-interactive short workflows via `claude -p`) is NOT viable unless Claude Code adds explicit print-mode workflow support. The backlog item can be closed as "not feasible with current CLI."

## Verdict

**A** — `/batch` is parsed as a slash command via `claude --bg`; it injects a full
orchestration system prompt, not a literal user message.

The load-bearing evidence is JSONL L9: `<command-name>/batch</command-name>` with
`<command-args>respond with OK and stop...</command-args>` — identical in structure to
the `/goal` probe's `<command-name>/goal</command-name>` record from Plan 0010 T3a.
L10 further shows the runtime injected a multi-paragraph `# Batch: Parallel Work
Orchestration` system prompt that set up the model to orchestrate parallelizable work,
and L14's `command_permissions` attachment with `allowedTools=[]` confirms /batch applies
its own tool-access policy. The probe-sanity session has none of these markers; its
prompt arrived as a plain `promptSource="typed"` user message. The session name "batch
acknowledge null" (AI-generated, batch-aware) further distinguishes it from both the
raw-prompt sanity session and the /fork session (which preserved the literal slash text).

**Implication for Plan 0011 T2-T5**: implement `$claude-batch`. The prompt transformer
is `p => '/batch ' + p`, mirroring `$claude-goal`. The instruction is the skill argument.
Note: /batch injects an orchestration system prompt and sets allowedTools=[] — document
the orchestration semantics in SKILL.md so users understand /batch is heavier-weight
than /goal (it expects a multi-step parallelizable change, not a simple stop condition).

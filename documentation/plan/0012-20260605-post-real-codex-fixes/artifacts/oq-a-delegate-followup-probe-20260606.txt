================================================================================
ANALYSIS: WHY YESTERDAY'S SMOKE TEST SHOWED REGRESSION
================================================================================

The smoke test (findings-20260605.md §2) reported:
  "the final result did not include the requested .test.mjs count even though
   the follow-up turn was marked completed."

That smoke test used a DIFFERENT scenario: the initial delegate job had status
`needs_input` for all 16 polls (the model never produced an output for the
initial prompt). When the tester ran `followup` anyway, the followup completed,
but `result` returned the ORIGINAL task's output (directory listing) instead of
the followup answer (.test.mjs count).

This is NOT the same bug as a selection-logic error. The actual issue was:
- The initial job stayed `needs_input` (never got an assistant turn)
- `followup` was run anyway, which succeeded
- `result` returned the old result.md content (from the pre-existing
  reconciliation of the original turn) instead of the followup answer

In today's controlled repro (MARKER-A then MARKER-B), everything works
correctly because:
- The initial delegate completed normally (MARKER-A)
- The followup was sent to an awaiting_followup job (correct state)
- The reconciler overwrote result.md with MARKER-B

The smoke-test failure was a **scenario interaction**: a needs_input initial job
combined with a followup that used the old result path. The result.md overwrite
mechanism works; the issue is the pre-condition check for followup eligibility
vs. what `result` reports.

VERDICT
================================================================================

VERDICT C — both result paths are healthy when used in the intended flow.

Justification: The controlled repro with MARKER-A/MARKER-B shows correct
behavior at every step. The smoke test regression was caused by the initial
job being stuck in needs_input (never producing a result), then followup
being run in that state — a different execution path than the one probed
today. The result.md single-file-overwrite pattern is working as designed
for the follow-up flow.

Root cause of the smoke-test observation: The delegate job stayed needs_input
(model never replied to the initial prompt), making the initial result.md empty
or absent. After followup, result.md was written with the followup's output —
but `cmdResult` reads the file and the reconciler may have been reading stale
turn[0] data from a cached job record rather than the freshly updated turn[1].
The reconcile inside sendFollowupTurn should ensure freshness; in the smoke test
the job was in an abnormal state (needs_input → followup) that may not have been
fully reconciled.

What would unblock a future verdict (if the bug recurs):
- Capture the raw job_<id>.json before and after the followup
- Verify whether turn[0].result.path differs from turn[1].result.path
- Check if cmdResult reads job.result.finalMessagePath (top-level) rather than
  the latest turn's result path

plausible in the smoke-test scenario.

================================================================================
ROOT CAUSE ANALYSIS
================================================================================

BEHAVIOR OF claude --bg --help:
When run in a shell with a TTY, `claude --bg --help` completes in ~0.5s and
prints a "backgrounded · <shortId> (idle — send a prompt to start)" message.
It CREATES an idle background session as a side effect (confirmed by 8 idle
sessions accumulating during probing; 3 created by these probes).

The critical behavior difference:
- With TTY:        claude --bg responds immediately (0.5s), writing session info to stdout
- Without TTY (piped stdio, spawnSync/spawn shell:false):
    claude --bg does NOT flush any output within 5000ms → SIGTERM timeout

The probe comment says "Real Claude prints usage for --bg" — but this is WRONG
as of claude 2.1.166. `claude --bg --help` does NOT print help text; it
CREATES a backgrounded idle session and prints the session handle. The `--help`
flag is silently ignored when combined with `--bg`.

Under piped stdio (no TTY), the session creation appears to block or delay
output beyond the 5s DEFAULT_TIMEOUT_MS. The likely mechanism:
  - claude --bg communicates with a daemon process over a socket/IPC channel
  - The daemon handshake or session creation writes to the terminal, not stdout
  - With piped stdio, the write buffer is not flushed within 5s
  - OR: the daemon path detects no-TTY and defers the "backgrounded" output

The Codex sandbox runs dispatcher commands as child processes with piped stdio
(no TTY). This is exactly Probe 4's shape: spawn with shell:false, piped stdio,
5s timeout. Result: SIGTERM at 5006ms.

WHAT THE PROBE IS ACTUALLY TESTING:
The probe was designed to test "claude --bg command shape is recognized" but
the actual behavior is "claude --bg creates an idle session". The probe
produces a false positive in TTY contexts (it succeeds) and a false negative
in no-TTY contexts (it times out). The probe is testing the wrong thing.

CORRECT ALTERNATIVE PROBE SHAPE:
The probe should not run `claude --bg` at all. Instead it should test for
the presence of --bg support without spawning an idle session. Options:

  Option A (recommended): Test daemon availability via `claude daemon status`
    - Already probed separately; always fast
    - Indirectly confirms --bg sessions can be created

  Option B: Parse `claude --version` and check version floor
    - claude --version is fast (42ms) and TTY-independent
    - --bg available >= 2.1.149 per existing version-floor logic

  Option C: `claude agents --json` (already probed as claude-agents-json)
    - Confirms agent infrastructure is running
    - Fast and TTY-independent

  Option D: Treat claude-bg-no-prompt as a version-floor check
    - Same approach as the opus-4-8-supported, workflows-supported, bg-exec-supported
      probes already in cmdSetup (lines 129-248 of claude-companion.mjs)
    - Zero subprocess invocation; pure version comparison

RECOMMENDED FIX:
Replace the `claude --bg --help` subprocess call with a version-floor check
(same pattern as bgExecProbe). If version >= 2.1.149 → ok. This eliminates
the TTY dependency entirely and doesn't create idle side-effect sessions.

Exact file + line range to fix:
  packages/runtime/src/doctor.ts:507-527 (probeClaudeBgNoPrompt function)
  packages/runtime/dist/doctor.js:421-446 (compiled output — must be rebuilt)

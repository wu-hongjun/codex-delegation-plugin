ANALYSIS
================================================================================

WHAT HAPPENED IN REPRO 1 (success):
- The /goal slash command was injected into a fresh bg session
- Claude processed the three user messages (command, stdout, stop-hook directive)
- Claude replied (entry [14]: ack), then used the Read tool, then answered
- Session reached awaiting_followup (Stop hook fired after goal was satisfied)
- result.md was written; cmdResult returned the answer

WHAT HAPPENED IN REPRO 2 (failure — matches smoke test):
- The /goal slash command was injected into a fresh bg session
- Three user messages were written to JSONL (same as repro 1)
- NO assistant reply appeared — the session sat idle
- Reconciler mapped the idle bg session to needs_input status
- The session never processed the injected prompt

COMPARISON OF THE TWO SESSIONS:
  Repro 1 (cf028bae): 43 JSONL entries — model executed normally
  Repro 2 (613865c4): 14 JSONL entries — model never responded

This is intermittent non-deterministic behavior in Claude Code's bg session
processing of injected /goal commands. The plugin's injection mechanism
(driver.send injects the /goal text into the session) appears to work — the
JSONL confirms entries [11-13] were written — but the model's async processing
of those entries is not guaranteed.

POSSIBLE ROOT CAUSES:
1. Race condition in bg session message delivery: the session may not have
   fully initialized before the prompt was injected, causing the message to
   be lost or ignored
2. Claude daemon's internal queue dropped the message during high load
3. The /goal command's Stop hook registration (entry [13]) sometimes fails
   to trigger the model to respond

CRITICAL OBSERVATION — Not a plugin-side sentinel detection bug:
The failing session's JSONL has NO assistant entries at all. This rules out
the hypothesis that "goal_status sentinel detection is brittle" (which would
require an assistant entry to be present but unparsed). The model simply
never replied. This is upstream behavior in the Claude bg session runtime.

YESTERDAY'S SMOKE TEST:
The smoke test ran the goal command and got needs_input for all 16 polls.
The JSONL for that session (73cd5d3f) was not examined here, but based on
today's evidence, the most likely explanation is the same: a fresh bg session
that received the /goal injection but the model never picked it up.

================================================================================
VERDICT
================================================================================

VERDICT C — upstream /goal brittleness in bg sessions; no plugin-side fix applicable.

Justification:
- Repro 1 demonstrates the goal flow works correctly when the bg session picks
  up the injected command (43 JSONL entries, correct result)
- Repro 2 demonstrates intermittent failure where the model never responds to
  the injected /goal prompt (14 JSONL entries, no assistant reply)
- The failure is non-deterministic: same prompt, same env, same claude version,
  different outcomes on consecutive runs
- The plugin's injection mechanism is correct (entries [11-13] written in both)
- The model's non-response is upstream behavior in the Claude bg daemon

WORKAROUND TO SHIP (per plan §4 T6 Verdict C guidance):
Update $claude-goal/SKILL.md to recommend:
  1. Use explicit, short, concrete directives — avoid multi-clause sentences
  2. Include a fallback polling pattern with a retry: if needs_input after
     first poll, stop and re-delegate (bg session failed to pick up the goal)
  3. Recommend: "Stop once you've answered." is sufficient; longer preambles
     may compete with the Stop hook directive in entry [13]

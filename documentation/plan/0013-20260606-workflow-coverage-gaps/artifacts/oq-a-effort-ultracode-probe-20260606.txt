SIDE EVIDENCE
-------------

1. claude --help explicitly lists valid --effort values: (low, medium, high, xhigh, max).
   "ultracode" is absent from this list.

2. When passed --effort ultracode, claude prints at startup:
   "Warning: Unknown --effort value 'ultracode' — ignoring it and using the default effort."
   The warning goes to stderr; the process exits 0 (not a fatal error).

3. The driver (packages/driver-claude-code/src/background-session.ts lines 101-102) passes effort
   directly: `argv.push('--effort', opts.effort)`. So the warning is emitted by claude itself, not
   by the driver.

4. JSONL comparison — A2 vs A1:
   - Both sessions produced structurally identical JSONL: session-start hooks, plain user turn,
     single assistant turn with end_turn stop.
   - No records distinguishable as workflow planning in either.
   - The only observable difference is slightly different token counts (5509 input_tokens vs 5406),
     consistent with normal session-to-session variance, not with any effort change.

5. The `/effort ultracode` TUI slash command is a separate code path (sets auto-orchestration mode
   for the session interactively). The CLI --effort flag maps to a different, static effort level
   enum that does NOT include "ultracode".


VERDICT
-------
A-partial

Justification:
  - The CLI flag does NOT reject "ultracode" with a non-zero exit (so Verdict B is incorrect).
  - The CLI flag does NOT activate auto-workflow planning when passed "ultracode" — the value is
    silently ignored with a warning and the default effort is used.
  - "ultracode" is not a valid value for --effort at all; it is only meaningful as a TUI slash
    command keyword (/effort ultracode) or as the ultracode: <prompt> prefix for $claude-workflow.
  - JSONL shows no distinguishing markers (no planning records, no phase tracking) in A2 vs A1.

Implication for T2:
  Verdict A-partial means `--effort ultracode` provides no capability beyond default effort. The
  recommended valid values for --effort are low, medium, high, xhigh, max. For T2, the implementation
  strategy is: document the valid --effort values (low/medium/high/xhigh/max) on existing skill
  SKILL.md files; note that "ultracode" is a TUI-only mode and the closest CLI equivalent is xhigh
  or max. No new $claude-effort skill is needed because the TUI /effort ultracode mode is not
  reachable via CLI flag; the existing $claude-workflow skill (which uses the ultracode: prefix
  transformer) is the correct skill for that use-case. T2 becomes a docs-only update.

Capture method: node-pty spawn 'claude' (bare), inject ultracode: prompt, approve

interactive TUI session. Background sessions started via cc workflow (--bg flag)

from within the same PTY session using the 'ultracode:' prompt prefix.

  4. Claude responds: "Running in background · /workflows to monitor and save"

SECTION 4: ANSI STRUCTURE SUMMARY
════════════════════════════════════════════════════════════════

Escape sequences observed in /workflows panel:
  \x1b[38;2;R;G;Bm     24-bit RGB foreground color (truecolor)
  \x1b[48;2;R;G;Bm     24-bit RGB background color
  \x1b[<col>G           Move cursor to absolute column (key for column layout)
  \x1b[<n>A/B/C/D       Relative cursor move (used for panel scroll-back)
  \x1b[K                Erase to end of line (used to clear blank rows below data)
  \x1b[1m / \x1b[22m    Bold on/off (workflow name in header)
  \x1b[3m / \x1b[23m    Italic on/off (footer "Esc to close")
  \x1b[?2026l/h         Synchronized output protocol (atomic redraws)
  \x1b[?25h/l           Cursor show/hide

Column layout (list view row, 120-col terminal):
  Col  1- 2:  indent/padding (spaces)
  Col  3:     status symbol (◯ = running, expected: ✓ = complete, ✗ = failed)
  Col  5-24:  workflow slug (truncated if >19 chars)
  Col 25-100: description prose (wraps or truncates at col 100)
  Col 101-120: "N/M agents done · Xs"  (right-side metrics)

Key color meanings:
  rgb(0,204,204)   = teal     — panel header separator line
  rgb(153,153,153) = gray     — secondary text (description, nav hints, counts)
  rgb(177,185,249) = lavender — /workflows link hint in chat output
  rgb(78,186,101)  = green    — success bullet ⏺ in chat
  rgb(255,255,255) = white    — loading spinner ✢
  rgb(255,193,7)   = amber    — warning text (token usage warning)
  rgb(177,185,249) = blue     — selected row highlight (❯ indicator)

Stability assessment:
  - Column positions (via \x1b[<col>G) are FIXED at specific column numbers
    (col 25, col 101, col 92 for nav) regardless of content length
  - Workflow name is NOT padded — it ends where it ends, then cursor jumps to col 25
  - Re-renders are incremental: only changed cells updated (agent count ticks)
  - The panel does NOT use alternate screen (\x1b[?1049h) — it is inline
  - Two captures of the same session (0/0 then 0/3) show the row structure is stable

VERDICT
════════════════════════════════════════════════════════════════

Verdict: A-partial

Justification:
  The /workflows panel has a CLEAN, PARSEABLE structure for the list view. Column
  positions are fixed via \x1b[<col>G absolute positioning:
    - Workflow name at col 5
    - Description at col 25
    - Agent progress at col 101

  Parsing strategy: regex per row using column-anchored patterns after ANSI-strip
  (strip ANSI codes, split on \r\n, match lines with ◯/✓/✗ at col 3). Reliable.

  The "A-partial" (not full A) is for two reasons:
  1. The drill-in view (Enter on a row to see per-agent breakdown) was NOT fully
     captured. The panel's second level (per-agent rows with phase/tool details)
     requires a slower workflow that stays in the running state longer (the test
     workflows completed within 0s before Enter could be processed). The list view
     is fully captured and parseable; the agent-level drill-in is structurally
     plausible but empirically unverified.
  2. Session scope constraint: /workflows only shows current-session workflows.
     The $claude-workflows skill must spawn claude (bare) AND trigger a workflow
     within that same PTY session (or attach to a session that has active workflows).
     This is a significant architectural constraint beyond what attach.ts handles —
     attach.ts attaches to EXISTING background sessions, whereas $claude-workflows
     requires an interactive TUI session with in-session workflow execution.

  Given the clean list-view structure, T2 harness + parser CAN ship for the list
  view. Drill-in parser is deferred to a follow-up capture with a slower workflow.

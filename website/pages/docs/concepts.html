Concepts and architecture

Codex remains the operator while provider-native processes do delegated work.

One direction

This plugin lets Codex delegate to Claude Code, Google Antigravity, Pi, or Qwen Code. It does not let those tools drive Codex, and it is not a compatibility alias for the former cc plugin.

Four layers

  1. Codex skill: a delegate: skill interprets the user request.
  2. Dispatcher: one Node entry point validates flags and routes the command.
  3. Driver: a provider adapter starts and observes provider-native work.
  4. Job store: provider-neutral records persist status, paths, and results. 
Driver
├── probe()                 → capabilities
├── startSession(options)   → provider handle
├── send(session, input)    → another exact-conversation turn
├── watch(target)           → lifecycle events
├── status(session)         → normalized status
├── stop(session)           → termination
└── dispose()               → cleanup

Jobs are not session names

Every delegation creates a new plugin job ID. The job record can include a provider session or process identifier, but callers should use the plugin job ID for status, wait, result, stop, follow-up, and review. A repeated --name labels separate sessions; it does not resume an earlier job.

Provider transports

Claude Code

ClaudeBackgroundDriver starts a first-party claude --bg session, reads machine-readable agent state and transcripts, and uses PTY attach for follow-up or human permission handoff. It does not parse the interactive TUI and does not fall back to claude -p.

Google Antigravity

AgyCliDriver owns a detached PTY supervisor around agy --prompt-interactive. A bundled Antigravity plugin publishes lifecycle and transcript metadata; the driver normalizes those records while retaining a raw ANSI log for native attachment. The exact UUID is job-scoped and global agy --continue state is never used.

Pi and Qwen Code

PiCliDriver supervises the oh-my-pi omp executable in JSON mode, while QwenCodeDriver supervises qwen in stream-JSON mode. Both capture a job-scoped session ID for exact --resume follow-up. These transports are headless, so permission prompts cannot be answered through terminal attachment.

Lifecycle states

queued / starting
The plugin has accepted the job and is establishing the provider process.
running
The provider is actively working.
needs_input
A permission or operator decision blocks progress.
awaiting_followup
A provider turn finished and its live session remains available for another instruction.
completed
The job has a final result. A still-live provider session can accept a targeted follow-up.
failed
The provider or supervisor exited unsuccessfully.
stopped
An operator terminated the job.
orphaned
The stored record exists but its provider process can no longer be reconciled.

State and artifacts

The default root is ~/.codex/codex-delegation-plugin/:

  • jobs/ stores normalized job records.
  • logs/ and result artifacts preserve provider output.
  • acks/ records privacy acknowledgement per workspace and provider.

Claude-owned transcripts and sidecars remain under Claude Code’s own directories. Plugin uninstall removes registration and marketplace pointers, not these job records or provider transcripts.

Provider follow-up models

A completed Claude turn is normally shown as awaiting_followup for 30 minutes after the last completed activity. The window is a status and UX policy, not a hard capability cutoff: an explicit follow-up may still attach if the Claude session remains live.

Antigravity keeps one persistent supervised TUI per job. A later follow-up is pasted into that same process, while $agy-attach proxies a user-owned terminal to native permission, task, conversation-fork, and subagent controls. Ctrl+] detaches without stopping it.

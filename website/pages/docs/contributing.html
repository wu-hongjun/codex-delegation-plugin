Contributing and releasing

Develop against the local marketplace, keep the packaged payload synchronized, and test every lane.

Local checkout

git clone https://github.com/wu-hongjun/codex-delegation-plugin.git
cd codex-delegation-plugin
npm ci
npm run build
npm test

Install the contributor marketplace from the checkout:

codex plugin marketplace add "$(pwd)/marketplace"
codex plugin add "delegate@codex-delegation-plugin-local"
codex plugin list

Refresh a local install

codex plugin remove "delegate@codex-delegation-plugin-local"
codex plugin add "delegate@codex-delegation-plugin-local"
codex plugin list

If the checkout moved, also remove and re-add the local marketplace pointer before reinstalling. Restart Codex after changing skills or plugin tools.

Repository validation

npm run lint
npm run typecheck
npm run format
npm test
npm run test:attach
npm run test:bench
node tools/package-marketplace.mjs --check
npm run site:test

The site test builds all pages, checks semantic landmarks, internal links, and the shared design-system stylesheet, rejects inline style and client JavaScript, verifies the 24-skill catalog against source, and syntax-smokes all 21 dispatcher commands.

Package the plugin payload

Source changes live under packages/. Regenerate the committed marketplace copy: 

node tools/package-marketplace.mjs --write
node tools/package-marketplace.mjs --check

Normal npm installation is for contributor dependencies only. The deployed product is the committed marketplace/plugins/delegate/ tree installed through Codex.

Documentation site

npm run site:build
npm run site:check
npm run site:serve

Open http://127.0.0.1:4173/codex-delegation-plugin/. Source fragments live under website/pages/; shared metadata and command catalogs live under website/; the generated _site/ directory is ignored. GitHub Actions deploys the verified artifact to Pages from main.

Design-system integration

Tailwind CSS 4 compiles the real tokens.css, styles.css, and prose.css from the pinned vvver-design-system submodule. The build also self-hosts its Switzer fonts. A small host layer supplies only the plugin palette and site-specific composition; the deployed pages need no client JavaScript or React runtime.

The submodule pins an exact reviewed commit rather than following moving main. The Pages workflow uses the dedicated read-only DESIGN_SYSTEM_DEPLOY_KEY Actions secret to check out the private repository. Update the gitlink deliberately when adopting a newer design-system revision. Pull-request CI remains secret-free; Pages runs the complete site verification before every deployment.

If the site later needs stateful design-system components, choose a React-capable static export. Keep the current semantic content fragments and command validation independent of presentation.

Release outline

  1. Make and test source changes.
  2. Regenerate and check the marketplace payload.
  3. Bump the plugin manifest version.
  4. Run repository, attachment, benchmark, site, and marketplace validation.
  5. Commit, tag, publish the GitHub release, and verify the public install.

Follow the repository’s complete release checklist.
